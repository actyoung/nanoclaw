## 消息流通架构

```
┌──────────┐     WebSocket/HTTP      ┌──────────┐
│  Feishu  │ ◄──────────────────────►│ Channel  │
│  Server  │                         │  Adapter │
└──────────┘                         └────┬─────┘
                                          │
                                          ▼
┌─────────────────────────────────────────────────────────┐
│  1. onMessage() 回调                                     │
│     - 存储到 SQLite (messages 表)                        │
│     - 标记 source_channel (feishu/cli/telegram)         │
│     - 调用 notifyNewMessage() 触发即时处理               │
│                                                          │
│  2. Message Loop (500ms 轮询 + 即时触发)                  │
│     - getNewMessages() 获取增量消息                      │
│     - 按 chat_jid 分组                                   │
│     - 检查 trigger (@提及 或 TRIGGER_PATTERN)            │
│                                                          │
│  3. GroupQueue 队列管理                                  │
│     - 检查并发限制 (MAX_CONCURRENT_CONTAINERS=5)         │
│     - enqueueMessageCheck() 入队                        │
│     - 或 sendMessage() 直接发送到活跃容器                │
│                                                          │
│  4. processGroupMessages()                               │
│     - 获取未处理消息 (getMessagesSince)                  │
│     - formatMessages() 格式化为 XML                      │
│     - runContainerAgent() 启动容器                       │
│                                                          │
│  5. 输出路由 (基于 source_channel)                        │
│     ┌─────────────────┬────────────────────────────────┐│
│     │ source_channel  │  行为                          ││
│     ├─────────────────┼────────────────────────────────┤│
│     │ feishu          │  reply → Feishu 频道           ││
│     │ cli             │  reply → CLI (不发送到频道)     ││
│     │ (其他)          │  reply → 对应频道              ││
│     └─────────────────┴────────────────────────────────┘│
└─────────────────────────────────────────────────────────┘
```

### 关键流程说明

1. **消息接收**: Channel Adapter (如 FeishuChannel) 接收外部消息，转换为标准 `NewMessage` 格式
2. **消息存储**: 存入 SQLite，触发 `notifyNewMessage()` 立即处理
3. **触发检测**: 检查 `@提及` 或 `TRIGGER_PATTERN`，非主群需要触发词才响应
4. **队列管理**: GroupQueue 管理并发，支持消息管道和排队等待
5. **容器执行**: 启动 Docker 容器运行 Agent，处理消息后输出
6. **路由回复**: 根据 `source_channel` 决定回复到哪个渠道

## 定时任务架构

```
┌──────────────────────────────────────────────────────────────┐
│                    Task Scheduler 调度器                      │
│                    (轮询间隔: 60秒)                           │
└────────────────────┬─────────────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────────────────┐
│  1. getDueTasks() - 查询 scheduled_tasks 表中 next_run <= now │
│     且 status = 'active' 的任务                               │
└────────────────────┬─────────────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────────────────┐
│  2. queue.enqueueTask(chat_jid, task_id, runTask)             │
│     - 检查是否已排队/运行中                                    │
│     - 任务优先级高于消息                                       │
│     - 如果容器忙，排队等待                                     │
└────────────────────┬─────────────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────────────────┐
│  3. runTask() - 执行任务                                       │
│     • 启动容器 (同消息处理流程)                                │
│     • isScheduledTask = true (10秒后自动关闭容器)              │
│     • context_mode: 'group' 复用会话 / 'isolated' 新会话       │
│     • 流式输出转发给用户                                       │
└────────────────────┬─────────────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────────────────┐
│  4. 任务完成后处理                                             │
│     • 成功: computeNextRun() 计算下次执行时间                  │
│     • 失败: classifyError() 分类错误                          │
│              - permanent: 标记 failed                         │
│              - retryable: 指数退避重试 (最多10次)               │
└──────────────────────────────────────────────────────────────┘
```

### 调度类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `cron` | Cron 表达式 | `0 9 * * 1-5` (工作日 9点) |
| `interval` | 间隔毫秒 | `3600000` (每小时) |
| `once` | 单次执行 | ISO 8601 时间戳 |

### 错误分类

- **Retryable**: 网络超时、Docker 暂时不可用、资源压力等，指数退避重试
- **Permanent**: 镜像不存在、权限拒绝、配置错误等，立即标记失败

### 容器镜像架构

项目支持多架构容器镜像：

| 架构 | Dockerfile | 适用系统 |
|------|------------|----------|
| ARM64 | `Dockerfile.arm64` | Apple Silicon (M1/M2/M3/M4) |
| AMD64 | `Dockerfile.amd64` | Intel Mac / Linux x86_64 |

`build.sh` 自动检测系统架构并选择正确的 Dockerfile。

## 安全架构

```
┌────────────────────────────────────────────────────────────────┐
│                        安全分层模型                             │
├────────────────────────────────────────────────────────────────┤
│ Layer 1: 凭证隔离                                               │
│   • .env 文件绝不挂载到容器                                     │
│   • Credential Proxy (端口3001) 拦截所有 API 请求               │
│   • 容器内使用 placeholder 密钥，真实密钥由代理注入             │
├────────────────────────────────────────────────────────────────┤
│ Layer 2: 文件系统隔离                                           │
│   • 主群只读挂载项目根目录                                       │
│   • .env 文件被 /dev/null 覆盖 (shadow mount)                   │
│   • 各群独立的 .claude/ 和 .evomap/ 目录                        │
│   • 挂载白名单验证 (mount-allowlist.json)                       │
├────────────────────────────────────────────────────────────────┤
│ Layer 3: IPC 授权                                               │
│   • 每个群独立的 IPC 目录 (data/ipc/{folder}/)                  │
│   • 消息发送验证: 只能发送到同群的 chat_jid                     │
│   • 只有主群可以: refresh_groups, register_group                │
├────────────────────────────────────────────────────────────────┤
│ Layer 4: 群组隔离                                               │
│   • 各群独立的 SQLite 数据 (通过 group_folder 区分)             │
│   • 任务只能操作同群的任务                                       │
│   • Session 按 group_folder 隔离                               │
└────────────────────────────────────────────────────────────────┘
```

## 关键配置项

| 配置 | 默认值 | 说明 |
|------|--------|------|
| `POLL_INTERVAL` | 500ms | 消息轮询间隔 |
| `SCHEDULER_POLL_INTERVAL` | 60000ms | 定时任务轮询间隔 |
| `CONTAINER_TIMEOUT` | 1800000ms (30min) | 容器硬超时 |
| `IDLE_TIMEOUT` | 1800000ms (30min) | 容器空闲超时 |
| `MAX_CONCURRENT_CONTAINERS` | 5 | 最大并发容器数 |
| `MAX_RETRIES` | 5 (消息) / 10 (任务) | 最大重试次数 |
| `TRIGGER_PATTERN` | `@{ASSISTANT_NAME}` | 触发词模式 |
| `CREDENTIAL_PROXY_PORT` | 3001 | 凭证代理端口 |
| `IPC_POLL_INTERVAL` | 1000ms | IPC 文件轮询间隔 |

配置位置: `src/config.ts` 和环境变量

## 启动流程

```
1. checkCriticalDependencies()
   ├── 数据库可访问性检查
   ├── 目录可写性检查
   └── 磁盘空间检查 (>100MB)

2. ensureContainerSystemRunning()
   ├── 确保 Docker/Podman 运行中
   └── 清理孤儿容器

3. loadState()
   ├── 加载 lastTimestamp (全局消息游标)
   ├── 加载 lastAgentTimestamp (各组处理游标)
   ├── 加载 sessions (各组 Session ID)
   └── 加载 registeredGroups (注册群组)

4. ensureCliGroup()
   └── 自动注册 cli:main 群组

5. startCredentialProxy()
   └── 启动端口 3001 的 HTTP 代理

6. startIpcWatcher()
   └── 启动 IPC 文件监控

7. startSchedulerLoop()
   └── 启动定时任务调度器

8. startMessageLoop()
   └── 启动消息处理循环

9. initializeChannels()
   └── 异步连接所有配置的消息渠道
```
